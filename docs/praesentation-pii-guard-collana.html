PII Guard
Datenschutz-Filter für KI-Coding-Tools

Personenbezogene Daten erkennen, maskieren, auditieren. Lokal. Automatisch. ISO 27001-konform.
Pfeiltasten zur Navigation – Thomas Körting / b-imtec GmbH
Das Problem
Jeder Prompt an Claude, Cursor oder Copilot kann personenbezogene Daten enthalten. Namen, E-Mails, IBANs – alles geht ungeschützt an die API.
Entwickler arbeiten mit echten Daten. Kundennamen in SQL-Queries, E-Mail-Adressen in Bug-Reports, Telefonnummern in Test-Szenarien. All das landet bei jedem Prompt ungeschützt beim KI-Anbieter.
Live-Beispiel

Was sich ändert

Ohne PII Guard
Du tippst: "Optimiere die Query für Kunde Max Müller (max@firma.de) IBAN DE89 3704 0044 0532 0130 00" Claude sieht: Echte Daten. Ungeschützt. Beim Anbieter.
Mit PII Guard
Du tippst: "Optimiere die Query für Kunde Max Müller (max@firma.de)" Claude sieht: "Optimiere die Query für Kunde Hans Schmidt (hs@beispiel.de)" IBAN: GEBLOCKT

Konzept

Drei Schutzschichten

Schicht 0
Git – Secrets
Gitleaks als Pre-Commit Hook. Fängt API-Keys, Passwörter und Tokens bevor sie ins Repository gelangen.
Schicht 1
Prompt – PII
PII Guard als Claude Code Hook. Maskiert Namen, E-Mails und Kundendaten automatisch bevor der Prompt an die API geht.
Schicht 2
Dateien – Ausschluss
.claudeignore für ganze Dateien. Schließt .env, Credentials und sensible Konfigurationen komplett aus.

Kernfeature

Typerhaltende Substitution

Naiver Ansatz
  • Max Müller wird zu [PERSON_1]
  • max@firma.de wird zu [EMAIL_1]
  • München wird zu [CITY_1]
  • KI bekommt kaputte Daten
  • Ergebnisse sind unbrauchbar
PII Guard
  • Max Müller wird zu Hans Schmidt
  • max@firma.de wird zu hs@beispiel.de
  • München wird zu Freiburg
  • KI bekommt gültige Daten
  • Ergebnisse sind korrekt
Und zurück
Die KI-Antwort enthält Fake-Daten. PII Guard mappt sie lokal zurück auf die echten Daten. Das Mapping verlässt nie den Rechner.
Das Mapping wird pro Session in .pii-guard/session-map.json gespeichert. Atomares Schreiben, automatisches Cleanup, Windows-kompatibel.
Architektur

Zwei Betriebsmodi

Modus A
Lokale Installation (pip)
Python 3.11+, pip install, spaCy-Modelle lokal. Für Einzelentwickler oder Maschinen mit Python-Setup. Hook läuft als Subprocess.
Modus B – empfohlen
Docker-Daemon
Container mit vorinstallierten Modellen (~2 GB). Kein lokales Python nötig. Hook sendet HTTP an localhost:4141. Ein Befehl: pii-guard docker start
Beide Modi nutzen die gleiche Pipeline: Presidio (NER) + Faker (Substitution) + Audit-Logger. Der Docker-Modus ist für Team-Rollouts empfohlen.

Audit-Werkzeuge

Wenn der Auditor kommt

Report
Compliance-Report
pii-guard audit-report generiert einen strukturierten Report: Zusammenfassung, Aufschlüsselung nach Typ und Aktion, Confidence-Statistik, Allow-List. Als Markdown oder CSV.
Test
Wirksamkeitstest
pii-guard audit-test führt vordefinierte Testfälle durch. PASS/FAIL pro PII-Typ, False-Positive-Prüfung, Schwellenwert konfigurierbar. Wird automatisch ins Audit-Log geschrieben.
Export
CSV-Export
pii-guard audit-export exportiert das Log als CSV. Filterbar nach Zeitraum. Öffnet direkt in Excel. Log-Rotation nach Größe (MB) und Alter (Tage, Default: 365).

Onboarding

Installation in 4 Schritten

  • Docker Desktop installieren (falls nicht vorhanden)
  • pip install pii-guard und pii-guard init im Projekt
  • pii-guard docker start --build – Daemon starten
  • Claude Code Hook registrieren (JSON in settings.json)
Danach läuft PII Guard automatisch bei jedem Prompt. Kein manueller Schritt nötig. Die Config (.pii-guard.yaml) wird ins Repo eingecheckt – zentrale Regeln, lokale Ausführung.

Warum als Gruppe

Ehrliche Einschätzung
PII Guard ist kein Allheilmittel. Es erkennt bekannte PII-Muster, aber keine Kontexte. Und es schützt den Prompt-Kanal – nicht den gesamten Datenfluss.
NER-basierte Erkennung hat Grenzen: Ungewöhnliche Namen, Abkürzungen oder domänenspezifische PII können durchrutschen. Die Confidence-Schwelle ist konfigurierbar. Regelmäßige Wirksamkeitstests (pii-guard audit-test) machen Lücken sichtbar.
